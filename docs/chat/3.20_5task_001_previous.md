## 根本原因：不是不想装，是装不了 + 看不到真正的报错

### 1. Agent 被锁在沙箱里，几乎没有自主权

Agent 的**唯一动作**就是生成 Python 代码，然后在 Apptainer 容器（只读镜像）里执行。它：

- 没有 shell 访问权限
- 没有交互式终端
- 容器是**只读挂载**的，pip install 实际上大概率写不进去
- HuggingFace Hub 默认设置为 `HF_HUB_OFFLINE=1`（离线模式）

### 2. 系统提示"骗"了 Agent

系统 prompt 里写着：

> *"You may pip install ONLY lightweight packages with prebuilt wheels; if install fails or is slow, immediately fall back to available packages."*

Agent 被告知可以装，但容器环境实际上很难装成功。而且给了一个**固定的可用包列表**：

```
numpy, pandas, scikit-learn, statsmodels, xgboost, lightgbm, 
torch, torchvision, torch-geometric, bayesian-optimization, timm
```

注意：**没有 `transformers`、`datasets`、`accelerate`**。

### 3. 错误信息被截断，Agent 看不到真正的报错

这是最致命的问题。Yelp 任务的实际情况是：

- `transformers` 其实**已经装了**（conda 环境里有）
- Agent 写了 `from transformers import DistilRobertaTokenizer, ...`
- 但执行输出被截断成：
  ```
  Traceback (most recent call last):
    File "solution.py", line 42, in <module>
      from transformers import (
  Execution time: 6 seconds
  ```
- **真正的错误信息被砍掉了！** Agent 根本看不到是 `DistilRobertaTokenizer` 不存在，还是版本不对，还是别的什么
- Agent 只能猜："可能是 transformers 没装？"，然后写了 `try/except ImportError` + `pip install` 作为兜底
- 但由于 transformers 本身导入成功（只是子类不存在），`except` 分支永远不会触发

### 4. 没有跨步骤的"记忆"

这是最让人崩溃的设计缺陷：

| 步骤 | Agent 行为 | 结果 |
|---|---|---|
| Step 5 | 用 DistilRoBERTa + Trainer | 失败 |
| Step 8 | 用 DistilBERT + Trainer | 同样失败 |
| Step 12 | 再用 DistilRoBERTa | 还是失败 |
| Step 15 | 又用 Trainer | 又失败 |
| ... | ... | 重复 20 次 |

**每个新 draft 节点都是从零开始**，不知道之前哪些路径已经失败过。框架的 memory 模块只记录"尝试过哪些 idea"，但**不记录"哪些包不可用"、"哪些错误模式反复出现"**。

### 5. Greedy Search 的探索策略太保守

```
搜索策略: 5 个 draft → 然后以 50% 概率 debug，每个节点最多 debug 3 次
```

问题：
- 5 个 draft 可能生成 5 个**本质相同**的方案（都用 transformers），然后全部进入 debug 死循环
- Debug 只看上一步的代码和报错，**没有全局视野**
- 直到 Step 24 才"碰巧"探索到 TF-IDF + sklearn 这个完全不同的方向

---

## 总结：这不是模型的问题，是框架的问题

```
Agent 视角：
┌─────────────────────────────────────────┐
│  我看到 transformers 不在可用包列表      │
│  → 但系统说我可以 pip install           │
│  → 我加了 pip install 作为 fallback     │ 
│  → 但 transformers 其实已经装了          │
│  → 真正的错误我看不到（被截断了）         │
│  → 我只能反复猜测和重试                  │
│  → 而且我不记得上一轮试过什么            │
│  → 循环 20 次...                        │
└─────────────────────────────────────────┘
```

**需要修的不是模型，而是框架：**

1. **不要截断错误信息** — 至少保留最后的 Exception 行
2. **加入失败记忆** — 让 debug/draft 知道"transformers.Trainer 需要 accelerate>=0.26 但环境没有"
3. **available_packages 要和实际环境一致** — 现在列表和容器实际装的包对不上
4. **增加探索多样性** — 5 个 draft 应该有策略地覆盖不同技术路线，而不是生成 5 个 transformer 变体

## 上游 vs 本地 Fork 对比结果

### 核心发现：`available_packages` 的 11 包限制是**上游原有设计**，不是 fork 引入的

| 方面 | 上游 (GitHub) | 本地 Fork | 差异 |
|---|---|---|---|
| `available_packages` 列表 | 11 个包 | 11 个包 | **完全一致** |
| 容器实际安装的包 | ~120+ | ~120+ | **完全一致**（同一份 pip.requirements.txt） |
| Agent 包发现机制 | **无** | **无** | 都没有 |

**上游和本地 fork 都存在相同的 11包 vs 120包 脱节问题。**

---

### 关键差异在 Prompt 措辞

**上游 (mlebench aira_operators)** — 措辞更**宽松**：
> "You have access to a Python environment with 1 NVIDIA H200 GPU(s) and 24 CPUs available, and the following packages installed: {{packages}}. **If you need to, feel free to use additional libraries that fit the problem.**"

**本地 Fork (airsbench aira_operators)** — 措辞更**限制**：
> "Available Packages: {{packages}}. **You may install additional Python packages ONLY if they are lightweight and likely to have prebuilt wheels (no compiling). If installation fails or is slow, immediately fall back to an approach using only the available packages.**"

上游说的是 **"feel free to use additional libraries"（随便用别的库）**，而 fork 改成了 **"ONLY lightweight packages with prebuilt wheels"（只能装轻量包）**。这个改动让模型更加畏手畏脚。

---

### 上游的 aide_operators 更友好

上游的 aide 风格 prompt（原始 AIDE 项目的提示词）：
> "You have access to Python and the following packages (already installed): {{packages}}. **Feel free to use additional libraries that fit the problem.**"

这种表述暗示 agent **可以大胆尝试其他库**，而不是被 11 个包的列表束缚。

---

### 本地 Fork 独有的改动

1. **新增 `airsbench/` 算子配置** — 上游只有 `mlebench/` 的配置，fork 新增了整套 airsbench 配置
2. **动态硬件检测** — 新增 `get_compute_overview()` 函数，替换上游硬编码的 "1 H200, 24 CPUs"
3. **限制性 pip install 措辞** — 从 "feel free" 改为 "ONLY lightweight"
4. **去掉 DataParallel 要求** — "multi-GPU is optional; do not require DataParallel"
5. **去掉 5-Fold CV** — 改为 "use a validation split; avoid multi-fold CV unless it is lightweight"

---

### 根因与建议

问题根源在**上游设计**就存在 — 给 agent 看 11 个包的列表，但容器里实际有 120+ 个包，而且没有任何 `pip list` 的环境探索步骤。本地 fork 通过更限制性的措辞进一步加剧了这个问题。

**建议的修复方案（按优先级）**：

1. **扩充 `available_packages`** — 至少加上 `transformers`, `accelerate`, `datasets`, `nltk`, `scipy`, `matplotlib`, `optuna`, `pytorch-lightning` 等高频包
2. **恢复宽松措辞** — 用上游的 "feel free to use additional libraries" 替代限制性措辞
3. **加入环境探索步骤** — 在 Step 0 让 agent 执行 `pip list` 获取真实包列表，注入后续 prompt
4. **长期方案** — 从 `pip.requirements.txt` 自动生成 `available_packages`，确保两者永远同步
